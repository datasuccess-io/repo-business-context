## 1. Deep integration with agentic AI / MCP / tools

**Does it belong?**  
Yes. This is one of the main reasons the Repo Business Context should exist at all. It should be explicitly designed as the “world model” for your agents.

**How I’d think about it**

- Keep the concept **framework-agnostic** (MCP, custom tool systems, serverless agents, etc.), but:
  - Assume **there will be agents** running:
    - In the IDE (coding, refactoring, local decisions).
    - In CI (reviewing PRs, proposing doc updates).
    - Possibly on a schedule (health checks, roadmap alignment).
- Treat `/context` as the **primary context source** for:
  - Defining tools and their allowed behaviors.
  - Explaining how tools should be used to serve business goals.

**Concrete integrations**

- In `AGENTS.md` (root or `/context/AGENTS.md`):

  - Explicitly state that any agent (IDE, CI, MCP-based) must:
    1.  Read `/context/CONTEXT.md` first.
    2.  Follow links to the relevant sub-docs before acting.
    3.  Prefer updating `/context` when business/tech reality changes.

- In `/context/operations` or `/context/tech` (see section 8 below):

  - Add a file like `tools-and-integrations.md` or fold this into `providers.md`:
    - List external APIs, MCP servers, and internal tools.
    - For each: purpose, safety constraints, and context docs it depends on.

**Verdict:** Definitely part of the concept, but phrase it generically so you’re not locked into one agent framework.

## 2. YAML front-matter (type, status, last-updated, etc.)

**Does it belong?**  
Yes, very aligned. It makes the context machine-readable and supports CI checks and agent behavior.

**Key benefits**

- Agents can quickly:
  - Filter for `status: active` vs `deprecated`.
  - See which docs are stale by `last_updated`.
  - Treat some docs as **authoritative**, others as **experimental**.
- CI can lint this metadata without parsing the full text.

**Potential downsides**

- Slight manual overhead to keep `last_updated` honest.
- For a solo founder, too many fields could become annoying.

**Recommendation**

Standardize a **minimal schema** and reuse it everywhere:

    ---
    context_type: business|persona|feature|architecture|data_model|decision|sprint|brand|agent
    status: draft|active|deprecated
    last_updated: 2026-01-11
    owner: founder
    ---

You can extend per file type (e.g., `sprint_date`, `version`), but keep the core fields the same.

**How to integrate**

- Define the schema once in `/context/CONTEXT.md`.
- In `AGENTS.md`, add rules like:
  - “When you edit a `/context` file, update `last_updated` to today.”
  - “If content is no longer accurate, set `status: deprecated` and suggest a new file.”

**Verdict:** Makes strong sense, but keep the schema small and consistent.

## 3. Enforcing updates via CI / PR checklists

**Does it belong?**  
Yes, as a **lightweight safety net**, especially since you’ll have agents proposing code changes.

**Keep it lightweight**

For a solo founder, heavy process is a risk. I’d keep it to:

- A simple `.github/pull_request_template.md` with a short checklist:

      - [ ] Have you considered whether `/context` needs an update?
      - [ ] If the change affects behavior visible to users, did you update:
        - [ ] `/context/product/features.md`
        - [ ] `/context/tech/domain-model.md` or `data-model.md` if relevant

- And a norm in `AGENTS.md`:

  > Before finalizing a PR, AI agents must scan `/context` and propose concrete doc updates when they detect feature or behavior changes.

**Verdict:** Yes, but keep it as a reminder + optional enforcement, not a hard blocker for everything.

## 4. GitHub Actions / workflows to support this

**Does it belong?**  
Yes, but again, carefully scoped. Two types of workflows seem worthwhile:

1.  **Lint /context docs**

    - Validate YAML front-matter is present and well-formed.
    - Optionally ensure `last_updated` isn’t wildly out of sync (e.g., more than N months old) for core docs like `vision.md`.

2.  **Context Steward trigger**

    - On a schedule (e.g., weekly) or on PRs, run an “agent” (using the OpenAI API or similar) that:
      - Reads diffs in code.
      - Checks `/context` docs touched by those areas.
      - Opens a PR or comment suggesting doc updates.

You don’t need to implement that now, but **document the intent** in `/context/operations/workflows.md` and `.github/README.md` (or similar), so future you (or agents) know this is the desired behavior.

**Verdict:** Makes sense as optional, pluggable infrastructure. Don’t make RBC depend on it, but design RBC so these workflows are easy to add.

## 5. Specialized “Context Steward” agent

**Does it belong?**  
This is very aligned with the idea. Think of it as the **guardian of the Repo Business Context**.

**Role**

- Periodically (or per PR) inspect:
  - Code / schema changes.
  - `/context` docs.
- Detect mismatches and either:
  - Directly open PRs to update `/context`.
  - Or leave review comments: “Feature X changed, but `/context/product/features.md` still describes the old behavior.”

**Where to define it**

- Add `/context/agents/context-steward.md`:

  - Purpose.
  - Inputs (git diff, list of changed files, current `/context` snapshot).
  - Rules (never invent business decisions, only infer from code + existing docs; always mark docs as `draft` if uncertain).
  - Expected outputs (PR with updated docs, or issue/PR comment with suggestions).

- Reference it in `AGENTS.md` as a **special role** agent that CI or local scripts can invoke.

**Verdict:** Yes, this is one of the most interesting parts of the concept.

## 7. Sprints follow deploys; rhythm is commits & deploys, not planned Scrum

**Does it belong?**  
Yes, and this is important enough that it should influence the structure.

Your reality:

- You talk to an LLM in the IDE.
- Pick something to build.
- Code → deploy → then retrospect.
- There’s a backlog of ideas, but “sprints” are effectively **build cycles** tied to deployments, not scheduled plans.

**Implications for `/sprints`**

Your current structure:

    sprints/
      active.md
      backlog.md
      20251201.md
      20250105.md

I’d adjust semantics:

- `backlog.md` → list of candidate tasks/ideas.
- `active.md` → what you are currently focusing on _right now_ (can be 100% driven by IDE chats).
- Historical files (e.g., `20260111.md`) → **retrospective logs** for each meaningful deployment or batch of changes.

You could rename the folder to reflect this better, e.g.:

- `/loops/` or `/shipping/` or `/cycles/`.

But if you like “sprints”, just define it clearly in `/context/CONTEXT.md`:

> In this repo, `sprints/` are **post-hoc snapshots of shipping cycles**, not pre-planned time-boxed iterations. Each sprint file is created after a meaningful deploy to document what changed, why, and what we learned.

**For agents**

- In `AGENTS.md`, you can instruct:

  - Use `sprints/backlog.md` as the main source of candidate work.
  - After completing and deploying a substantial change, propose:
    - Updates to `/context/product/features.md` etc.
    - A new `sprints/YYYYMMDD.md` entry summarizing that cycle.

**Verdict:** Yes, and it’s a distinguishing feature of the concept: **deploy-driven planning**, not calendar-driven.

## 8. Splitting `/tech` into something lighter

You’re right: `/tech` is overloaded. I’d split it into:

- `/tech/` → **System and domain**.
- `/development/` → **How we build**.
- `/operations/` → **How it runs in the real world**.

### Proposed split

Current list:

- `architecture.md`
- `domain-model.md`
- `data-model.md`
- `conventions.md`
- `constrains.md`
- `providers.md`
- `observability.md`
- `workflows.md`
- `decisions/`

Refactor to:

    /context
      /tech
        architecture.md        # High-level system diagram, components
        domain-model.md        # Concepts, business logic
        data-model.md          # Tables, schemas, storage specifics
      /development
        conventions.md         # Coding style, repo structure, testing patterns
        constraints.md         # Libraries/frameworks to avoid, performance/security constraints
        workflows.md           # Local dev workflow, branching model, CI usage
        decisions/
          20260111-chose-db-supabase.md
      /operations
        providers.md           # SaaS providers, APIs, MCP servers, deployment platform
        observability.md       # Logging, metrics, alerting strategy
        runbook.md             # How to handle incidents, common operational tasks
        deploy-log.md          # Optional high-level record of major deploys (can link to /sprints)

Notes:

- **`tech/`** stays compact and focused on the product’s mental model and structure.
- **`development/`** collects _developer-facing practices_ and history (ADRs).
- **`operations/`** is where you can later describe:
  - Where agents run (CI, serverless functions, etc.).
  - How deploys happen.
  - Which providers and tools are in play (including AI tools/MCP servers).

This split also fits nicely with your idea of advanced agents and CI:

- Agents that reason about code architecture read `/tech` + `/development`.
- Agents that reason about deployment, monitoring, or provider usage read `/operations`.

**Verdict:** Splitting improves clarity and keeps each directory’s purpose crisp.

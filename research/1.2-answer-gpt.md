## 1. Reframing the idea clearly

What you’re really defining is:

- A **folder inside the codebase**, e.g. `/meta/`, `/project/`, `/business/`, whatever name, which contains:

  - **Business docs**: problem, personas, positioning, use cases, pricing, etc.
  - **Tech docs**: stacks, architecture, domain model, data model, providers, observability, etc.
  - **Brand/comms docs**: voice, content principles, examples.
  - **Agent instructions**: how AI tools should behave, what to optimize for, how to interpret the rest of the docs.

- All:

  - Written in **English**.
  - In **Markdown**, structured for easy parsing.
  - **Living and versioned with the code**, updated along with changes in product and business.

- Primary goals:

  - Give AI agents **full context of both business and technology**, not just code.
  - Make it the **canonical reference** for any activity: coding, UX, marketing, docs, social posts, landing pages.
  - Reduce mental load for you as a solo founder (and later for teams) by encoding the “head of the founder” into a formal, queryable corpus.

That’s a sound and coherent problem statement.

## 2. Feasibility & necessity

### Feasibility

For a solo founder building a web app:

- **Technically simple**:
  - It’s just Markdown files in a folder. No infrastructure barrier.
  - Versioning and diffing comes “for free” via Git.
- **Operationally moderate**:
  - The main cost is **discipline**: keeping docs in sync and not letting them rot.
  - If you already think in text (notes, Notion, etc.), migrating that into the repo is straightforward.

For later teams:

- Very feasible if you define **ownership rules**:
  - Product owner updates business/roadmap/docs.
  - Tech lead/engineers update architecture/data/ADR-like docs.
  - Marketing updates brand/comms docs.
- You’d likely complement this with a wiki or knowledge base, but the repo-resident docs remain the **structured nucleus** for AI and automated tooling.

### Real necessity

Where it _is_ necessary / valuable:

- You want AI agents (in IDEs, CI, internal tools) to:

  - Make decisions aligned with **business goals**, not only code patterns.
  - Suggest features, fixes, or copy that match **positioning, persona, and constraints**.
  - Understand **domain language** and entities.

- You’re a solo founder juggling:

  - Product strategy
  - Tech decisions
  - Marketing & brand
  - Ops

  Encoding this in a consistent structure makes sense.

Where it might be overkill:

- Very tiny, throwaway apps / prototypes.
- Cases where business is unclear or chaotic, and you change direction weekly. (Though having a clear doc of “current hypothesis vs rejected ones” can still help.)

Overall: this _is_ solving a real, emerging pain: **AI tools lack application-specific, business-level context**, and that cripples their usefulness beyond trivial coding assistance.

## 3. Pros

**1. Single source of truth, in the same place as the system that enforces it (the code).**

- You remove the gap between “strategy in Notion” and “code in Git”.
- Branches can contain both **feature code + updated business/tech implications**.

**2. AI-first design.**

- If you write with:

  - Clear headings
  - Consistent terminology
  - Simple tables / lists
  - Explicit marking of current vs future

  …then agents can:

  - Generate tasks consistent with roadmap.
  - Produce UX copy in the right voice.
  - Respect non-functional constraints (performance, compliance, pricing, etc.).

**3. Better onboarding (including for your future self).**

- Future devs, marketers, or PMs can understand:
  - Why things exist (business reasoning)
  - How they exist (architecture)
  - For whom they exist (personas)

…without hopping through disconnected tools.

**4. Decision traceability.**

- If you include micro-ADR-style records (short decision notes), you get:
  - Quick recall of **why** you picked stack X or dropped feature Y.
  - Context for AI agents: “Don’t propose a mobile app; we explicitly decided web-only for now, see doc.”

**5. Facilitates multi-surface consistency.**

- Landing pages, in-app copy, onboarding flows, social posts — all can be made to:
  - Use the same value props.
  - Target the same persona narratives.
  - Use the same vocabulary.

Because there’s a single, well-structured source behind them.

## 4. Cons / risks

**1. Stale documentation.**

- Biggest risk: reality diverges from docs.
- Then AI agents (and humans) make wrong assumptions.
- Mitigation:
  - “No feature is done until docs in `/project/` are updated.”
  - CI checks (even lightweight, like `TODO` comments or a checklist).
  - Clear dating: each doc or section has a “Last updated: YYYY-MM-DD”.

**2. Overhead vs speed.**

- For a scrappy solo founder, stopping to update docs may feel like a drag.
- But if you keep scope small and concise, you can keep this to minutes per feature.

**3. Duplication with other tools.**

- If you also keep a Notion workspace, there’s risk of:
  - Two roadmaps.
  - Two sources of persona definitions.
- You’ll need to decide:
  - “The repo `/project` folder is the canonical source. External tools mirror it, not vice versa.”

**4. AI hallucinating or over-weighting old decisions.**

- If AI tools trust the docs more than reality (e.g., you pivoted pricing but forgot to update), you get bad outputs.
- Clear labeling of **deprecated/archived** sections, and periodic cleanup, is essential.

**5. Scope creep.**

- It can become a dumping ground for everything: random ideas, chat logs, etc.
- You’ll want clear boundaries on:
  - What belongs inside (canonical decisions & definitions).
  - What stays outside (raw notes, unfiltered brainstorms).

## 5. What already exists, and how this is different

Related things:

- **README.md**: high-level overview, install instructions. Not business-rich or AI-oriented.
- **VS Code `instructions.md` / editor configs**: instructions for AI code tools, but usually **narrow to coding behaviors**.
- **Architecture docs / ADRs**: technical decisions, but rarely include business model, personas, and brand voice.
- **Business Model Canvas, Lean Canvas, PRDs**: great for business view, but:
  - Often sit in separate tools (Notion, PDFs).
  - Not optimized for AI consumption or tied to code.
- **Internal wikis / Notion / Confluence**: flexible, but:
  - Not versioned with code.
  - Harder to use as strict canonical source for tools that run inside repos.
- **Developer portals (e.g., Backstage)**: centralize services and docs, but:
  - Heavyweight.
  - Usually infra/dev-focused, not marketing/brand/persona-focused.
  - Not inherently AI-first.

Your twist:

- **Same repo as code + explicitly designed for AI** (structure, conventions).
- **Business + tech + comms in one small, structured corpus**.
- **Actively intended as the main “brain” for agents** automating tasks across code, content, and ops.

That’s a distinct enough niche to be meaningful.

## 6. Refinement: concrete structure and conventions

You don’t want 50 files. You want a **small, opinionated schema**.

### Example folder structure

At repo root:

    /project
      /business
        vision.md
        personas.md
        value-prop.md
        pricing-and-model.md
        use-cases.md
      /product
        roadmap.md
        features.md
        ux-principles.md
      /tech
        architecture.md
        domain-model.md
        data-model.md
        stack-and-tools.md
        observability.md
      /brand
        voice-and-tone.md
        messaging-guide.md
        examples.md
      /agents
        coding-agent.md
        content-agent.md
        operations-agent.md
      index.md

You can rename `/project` to whatever you like; core idea is the same.

### Purpose of key files

- `index.md`

  - High-level orientation: what this folder is, how it’s structured, how to interpret it.

- `/business/vision.md`

  - One page defining:
    - Problem
    - Who it’s for
    - What the product will & won’t do
    - North-star metrics or qualitative goals

- `/business/personas.md`

  - 2–3 primary personas:
    - Background, goals, pains
    - Tools they use
    - How they discover your app

- `/business/value-prop.md`

  - Core value statements.
  - Short, medium, and detailed versions.

- `/business/pricing-and-model.md`

  - Revenue model (SaaS tiers, usage, etc.).
  - Important constraints: minimum ARPU, CAC assumptions, etc.

- `/product/roadmap.md`

  - Near-term, mid-term, long-term.
  - Mark each item with status: `planned / in-progress / shipped / dropped`.

- `/product/features.md`

  - Canonical list of current features.
  - Each feature: name, description, target persona, key flows.

- `/tech/architecture.md`

  - High-level diagram (even just text).
  - Frontend, backend, services, integrations.

- `/tech/domain-model.md`

  - Definitions of core domain concepts/entities in business terms.

- `/tech/data-model.md`

  - Overview of key tables/entities, not full schema dump.
  - Enough for AI to generate sensible queries and reason about data.

- `/tech/stack-and-tools.md`

  - Frontend stack, backend stack, infra, CI/CD, third-party providers.

- `/brand/voice-and-tone.md` + `examples.md`

  - Do/don’t rules.
  - Real examples of messages, headings, social posts, microcopy.

- `/agents/*.md`

  - Specific instructions optimized for each type of agent:
    - **Coding agent**: how to prioritize refactors vs features, what tests to write, security/performance constraints, coding style if not fully enforced by linters.
    - **Content agent**: which docs to read for value props, personas, brand voice; how to adapt message per persona.
    - **Operations agent**: how to interpret metrics or alerts, which docs to consult for troubleshooting, how to propose experiments.

### Conventions for AI-readability

To make this truly AI-first:

- **Consistent headings and sections.**  
  Example:

      # Feature: Smart Onboarding Checklist

      ## Summary
      ...

      ## Target persona
      - id: founder_early_saas

      ## Business goal
      - Increase activation rate from X to Y

      ## Status
      - in-progress

      ## Dependencies
      - domain entities: Workspace, User, ChecklistTemplate

- **Use IDs and tags.**

  E.g., in `personas.md`:

      ## Persona: Early SaaS Founder
      - id: founder_early_saas
      - stage: pre-revenue / < $5k MRR
      - primary goals: ...

  Then other docs reference `founder_early_saas` by ID.

- **Mark current vs future clearly.**

      ## Current capabilities (as of 2026-01-10)

      ## Planned (not yet built)

- **Avoid fictional placeholders.**  
  If something isn’t decided, say:

      ## Pricing
      - status: undecided
      - current assumption: ...
      - do not use this in public-facing copy yet.

And define a simple rule for yourself:

- Whenever you:
  - Ship a **new feature** → update `features.md` and `roadmap.md`.
  - Change a **stack choice** → update `stack-and-tools.md`.
  - Refine **positioning or persona** → update `vision.md`, `value-prop.md`, or `personas.md`.
  - Install/configure a new AI agent → make sure its instructions point to `/project` as primary context.

This is a compelling idea that addresses a real gap in modern AI-assisted development. Let me break down my evaluation:

## The Core Problem You're Solving

You're identifying that **context fragmentation** hurts AI-assisted development. Today:

- Business docs live in Notion/Google Docs (disconnected from code)
- Technical docs are often outdated READMEs
- AI agents operate with partial context, leading to misaligned outputs
- Solo founders must constantly re-explain context to AI tools

## Existing Solutions & Gaps

| Solution                                 | What it does                 | Gap                                             |
| ---------------------------------------- | ---------------------------- | ----------------------------------------------- |
| **Lean/Business Model Canvas**           | High-level business snapshot | No tech integration, not AI-readable in context |
| **README.md**                            | Project overview for humans  | Too shallow, no business model                  |
| **instructions.md / .cursorrules**       | AI coding behavior rules     | No business/product context                     |
| **PRDs (Product Requirement Docs)**      | Feature specifications       | Usually external, not living docs               |
| **ADRs (Architecture Decision Records)** | Tech decisions log           | No business reasoning connection                |

**Your gap is real**: None of these create a **unified, living knowledge base** that gives AI agents the full picture of _why_ the business exists, _what_ it's building, and _how_ it's built—all in one place, in the codebase.

## Feasibility Assessment

**Highly feasible.** It's just structured Markdown files in a folder. No special tooling required to start.

However, the **real challenge** is:

1. **Maintenance discipline** - Docs rot fast when separated from the work
2. **Granularity balance** - Too detailed = unmaintainable; too vague = useless to AI
3. **AI consumption** - Current AI context windows are limited; you need smart chunking

## Pros

- **Single source of truth** for human and AI collaboration
- **Reduces prompt engineering** - agents inherit context automatically
- **Decision traceability** - connects business "why" to code "what"
- **Onboarding accelerator** - new AI sessions (or humans) ramp up instantly
- **Version controlled** - business decisions tracked alongside code
- **Framework agnostic** - works with any AI tool that reads files

## Cons / Risks

- **Maintenance overhead** - another thing to keep updated
- **Scope creep** - temptation to document everything
- **No enforcement** - nothing stops code from drifting from docs
- **Context limits** - AI agents can't always load all docs at once
- **Learning curve** - defining what goes where takes iteration

## My Recommendations

### 1. Proposed Structure

```
/.context/                          # or /docs/context/, /.blueprint/
├── README.md                       # Index: what this folder is, how to use it
├── business/
│   ├── vision.md                   # Problem, mission, values, positioning
│   ├── model.md                    # Revenue streams, pricing, unit economics
│   ├── personas.md                 # ICPs, jobs-to-be-done, pain points
│   └── roadmap.md                  # Phases, priorities, what's in/out of scope
├── product/
│   ├── features.md                 # Current features, status, specs
│   ├── ux-principles.md            # Design system rules, tone, patterns
│   └── integrations.md             # Third-party services, APIs, webhooks
├── tech/
│   ├── architecture.md             # Stack, infrastructure, data flow
│   ├── data-model.md               # Entities, relationships, schemas
│   ├── conventions.md              # Code style, naming, folder structure
│   └── decisions/                  # ADRs (Architecture Decision Records)
│       └── 001-chose-nextjs.md
├── brand/
│   ├── voice.md                    # Tone, vocabulary, messaging guidelines
│   ├── assets.md                   # Logo usage, colors, links to assets
│   └── channels.md                 # Social media handles, content strategy
└── operations/
    ├── providers.md                # SaaS tools, credentials location, purposes
    └── workflows.md                # CI/CD, deployment, monitoring
```

### 2. Key Principles for Success

1. **Link, don't duplicate** - Reference external tools (Figma, Linear) rather than copying
2. **Use front-matter** - YAML headers help AI parse structure:
   ```yaml
   ---
   type: persona
   status: validated
   last-updated: 2026-01-10
   ---
   ```
3. **Keep files focused** - One concern per file, <500 lines ideally
4. **Create an AI-specific index** - A `CONTEXT.md` that tells agents which files to read for which tasks
5. **Automate where possible** - Generate `data-model.md` from actual schema, keep them synced

### 3. What Makes This Different

Call it something distinct. Ideas:

- **Project Blueprint** (`/.blueprint/`)
- **Context Layer** (`/.context/`)
- **Living Spec** (`/.spec/`)

This differentiates it from READMEs and instructions files—it's the **semantic layer** of your business.
